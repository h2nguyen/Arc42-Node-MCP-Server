= 4. Solution Strategy

== 4.1 Technology Decisions

[cols="2,2,5", options="header"]
|===
| Decision | Choice | Rationale

| *Runtime*
| Node.js 24+
| Modern JavaScript/TypeScript support, excellent async I/O, npm ecosystem

| *Language*
| TypeScript 5.7
| Type safety, better tooling, self-documenting code

| *Protocol*
| MCP (Model Context Protocol)
| Industry standard for AI tool integration, supported by major AI assistants

| *Transport*
| STDIO
| Simple, universal, no network complexity for local execution

| *Validation*
| Zod
| Runtime type validation with TypeScript inference

| *Documentation Format*
| Markdown & AsciiDoc
| Both formats supported; AsciiDoc default for professional docs

| *Template Source*
| Git Submodule
| Keep template in sync with upstream, version tracking

| *Format Architecture*
| Strategy + Plugin Pattern
| Extensible format support with native templates per language
|===

== 4.2 Top-Level Decomposition

The system follows a *modular tool-based architecture*:

[plantuml, top-level-decomposition, svg]
----
@startuml
!theme plain
skinparam backgroundColor transparent
skinparam componentStyle rectangle

package "Entry Point" {
    component "src/index.ts" as Entry
}

package "Arc42MCPServer" {
    component "src/server.ts" as Server
    note right of Server
      • MCP server initialization
      • Tool registration
      • Transport management
    end note
}

package "Modules" {
    component "Tools\n(src/tools/)" as Tools
    component "Templates\n(src/templates/)" as Templates
    component "Types\n(src/types.ts)" as Types
}

Entry --> Server
Server --> Tools
Server --> Templates
Server --> Types
Tools --> Templates
Tools --> Types
Templates --> Types
@enduml
----

== 4.4 Format Support Strategy

The system supports multiple output formats through a layered architecture:

[plantuml, format-architecture, svg]
----
@startuml
!theme plain
skinparam backgroundColor transparent

package "OutputFormatStrategy Interface" {
    rectangle "MarkdownFormatStrategy" as MD
    rectangle "AsciiDocFormatStrategy" as ADOC
}

note right of MD
  formatHeading()
  formatTable()
  formatCode()
  fileExtension: .md
end note

note right of ADOC
  formatHeading()
  formatTable()
  formatCode()
  fileExtension: .adoc
end note

package "FormatTemplatePlugin Interface" {
    rectangle "Per Language (EN, DE, etc.)" as Lang
    component "markdownPlugin\n(templates-md.ts)" as MDP
    component "asciidocPlugin\n(templates-adoc.ts)" as ADP
}

MD -[hidden]-> MDP
ADOC -[hidden]-> ADP
@enduml
----

[cols="2,4", options="header"]
|===
| Approach | Implementation

| *Native Templates*
| Each language has native AsciiDoc templates from vendor sources

| *Markdown Templates*
| Generated from arc42 content with proper Markdown syntax

| *Plugin Architecture*
| `FormatTemplatePlugin` interface for extensible format support

| *Factory Pattern*
| `createLanguageStrategy()` creates strategies with format plugins

| *Fallback Support*
| `createFormatPluginsWithFallback()` for missing format templates

| *Format Detection*
| Auto-detect format from file extension or config.yaml
|===
